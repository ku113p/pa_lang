## Per-Category Analysis

### Scanning/Group Kernels (PA's strength)
- Source ratio: 30-38% — PA source is less than half the size of x86 asm
- Bytecode ratio: 40-49% — PA bytecode is roughly half x86 machine code
- PA's `gld`/`gcm`/`gcp` compress x86 SIMD setup (movd + punpcklbw + pshuflw + punpcklqdq = 4 instructions) into 1 PA instruction (`gcm pF`)
- Wasm SIMD instructions pay a 0xFD prefix byte, making them 2-3 bytes each vs PA's fixed 2 bytes

### Scalar Kernels (moderate advantage)
- Source ratio: 41-45% — shorter mnemonics + cells still compress vs x86
- Bytecode ratio: 69-81% — closer to parity; scalar x86 instructions are already compact (2-3 bytes)
- PA and x86 have similar instruction counts for these kernels

### Negative Cases (PA advantage diminished)
- **Fibonacci** (source 57%, bytecode 92%): pure scalar arithmetic with no group ops. The cell table lacks `r0:=r3`, forcing an extended `mv!` instruction. PA bytecode nearly matches x86 in size.
- **Min byte** (source 66%, bytecode 92%): PA's condition flag (zero/nonzero) cannot express ordered comparison. Requires extended `cm!` instruction. x86's `cmp` + `cmovb` is extremely compact (5 bytes for compare-and-conditional-move vs PA's 6 bytes for compare + branch + move).

### Key Observations
- PA's advantage is strongest when group/predicate operations replace multi-instruction SIMD patterns
- On pure scalar code, PA offers modest source compression but near-parity bytecode size
- Extended instructions (3 bytes) appear in 4 of 7 kernels but represent only a small fraction of instructions
- PA vs Wasm is consistently strong (~32%) because Wasm's stack machine requires explicit `local.get`/`local.set` pairs (2 bytes each) that PA's cell system avoids
